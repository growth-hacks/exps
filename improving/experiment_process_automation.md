# Возможности автоматизации процесса экспериментов
**Создан:** 24 марта 2026
**Контекст:** Анализ полного корпоративного флоу из 13 шагов; выявлены зоны ручной работы, которые можно автоматизировать агентами

---

## Полный флоу эксперимента (13 шагов)

1. **Инициация** — заказчик заводит задачу. Выход: гипотезы, аудитория, метрики, user flow.
2. **Проработка вариантов** — черновые → финальные лендинги.
3. **Ревью и правки** — утверждённые варианты.
4. **Подготовка аудитории** (параллельно) — подтверждённый объём.
5.1. **Jira-задачи** (БЛОКИРУЮЩИЙ) — создаются задачи на отдельной доске, отдельно Android и iOS.
     В задаче: описание эксперимента, варианты, ссылки на лендинги, размер аудитории, placement, платформы.
     Статусы: Создано → Готово к выкату → Внедрено → Мониторинг → Закрыто.
5.2. **База знаний** (БЛОКИРУЮЩИЙ) — статья для колл-центра по шаблону: описание эксперимента, что видит клиент, возможные вопросы/ответы. Без этого запуск невозможен.
6. **Запуск** — Jira → «Готово к выкату»; саппорт выкатывает, ставит «Внедрено».
7. **Проверка после выката** — пользователь проверяет работу, Jira → «Мониторинг».
8. **Проверка трекинга** — GA4 события, внутренние дашборды, Excel/Google Sheets.
9. **Проведение теста** — ~1 неделя в проде.
10. **Завершение** — «можно удалять»; саппорт удаляет виджеты, Jira → «Закрыто».
11. **Анализ данных** — аналитик считает воронку, конверсии, стат. значимость.
12. **Презентация итогов** — результаты, выводы, рекомендации.
13. **Месячный отчёт** — единый шаблон, каждый Growth презентует свои эксперименты.

## Группа А — Немедленные wins (блокирующие, чистый текст)

### 5.1 Агент: growth-jira-task-generator

**Что делает:** По готовому плану и лендингам генерирует заполненные шаблоны Jira-задач — отдельно для Android и iOS.

**Входные данные:**
- `{ID}-plan.md` — описание, варианты, аудитория, placement, платформы
- Список ссылок на опубликованные лендинги (передаёт пользователь после публикации в Тильду)
- Размер аудитории (передаёт пользователь)

**Выходной артефакт:** `{ID}-jira-tasks.md`

**Структура задачи:**
```
Название: [краткое название эксперимента] — Android / iOS
Что делаем: [описание эксперимента из плана]
Варианты: A — [посыл], B — [посыл], ...
Ссылки на лендинги: Вариант A: [URL], Вариант B: [URL]
Размер аудитории: [N] пользователей
Placement: [главный экран / пуш / баннер]
Платформа: Android / iOS
```

**Почему ценно:** Блокирующий шаг перед запуском, сейчас полностью ручной. Агент заполняет всё из уже готовых артефактов пайплайна.

---

### 5.2 Агент: growth-kb-article-writer

**Что делает:** Пишет статью в базу знаний для колл-центра по стандартному шаблону.

**Входные данные:**
- `{ID}-plan.md` — гипотезы, аудитория, флоу, варианты
- `{ID}-copy-variant-{N}.md` — тексты лендингов (что видит клиент)

**Выходной артефакт:** `{ID}-kb-article.md`

**Структура статьи:**
```
Название эксперимента: ...
Период проведения: ...
Что видит клиент: [описание каждого варианта]
Целевое действие: [нажать CTA-кнопку]
Возможные вопросы клиентов:
  Q: Почему я вижу такой экран?
  A: ...
  Q: Что означает эта кнопка?
  A: ...
  Q: Могу ли я отказаться?
  A: ...
```

**Почему ценно:** Блокирующий шаг — без статьи запуск невозможен. Полностью генерируется из готовых артефактов.

## Группа Б — Высокая ценность (не срочно)

### 12. Агент: growth-experiment-summary

**Что делает:** Готовит презентацию по итогам эксперимента.

**Входные данные:**
- `{ID}-plan.md` — гипотезы, метрики, варианты
- Результаты от аналитика (передаёт пользователь: конверсии, стат. значимость, воронка)

**Выходной артефакт:** `{ID}-summary-presentation.md`

**Структура:**
```
Эксперимент: [название и ID]
Гипотеза: [что проверяли]
Варианты: [A vs B vs ...]
Результаты: [таблица метрик]
Победитель: [вариант X, +N%, p-value]
Выводы: [что узнали]
Рекомендации: [внедрять / не внедрять / доработать]
Следующие шаги: [что тестировать дальше]
```

---

### 13. Агент: growth-monthly-report

**Что делает:** Собирает месячный отчёт по всем экспериментам.

**Входные данные:**
- Список `{ID}-summary-presentation.md` за месяц
- Шаблон отчёта команды (передаёт пользователь или берёт из references)

**Выходной артефакт:** `monthly-report-{YYYY-MM}.md`

## Группа В — Частичная автоматизация (чеклисты)

### 7–8. Агент: growth-post-launch-checklist

**Что делает:** Генерирует персональный чеклист проверки после выката на основе артефактов конкретного эксперимента.

**Входные данные:** `{ID}-plan.md`, `{ID}-qa-report.md`

**Выходной артефакт:** `{ID}-post-launch-checklist.md`

**Содержание:**
```
Проверить в GA4:
  □ Событие cta_click_{ID}_1_android — должно появиться в Realtime
  □ Событие cta_click_{ID}_2_android — должно появиться в Realtime

Проверить в приложении (Android):
  □ Открыть приложение под тестовым аккаунтом из группы A
  □ На главном экране должен быть виджет варианта A
  □ Клик на виджет → открывается лендинг варианта A
  □ CTA-кнопка ведёт на правильный экран

Jira:
  □ После подтверждения → перевести задачу в статус "Мониторинг"
```

## Группа Г — Публикация лендингов (инфраструктура)

Сейчас после каждого изменения HTML нужно вручную заходить в Tilda и вставлять файл.
Tilda не имеет write API — обновить содержимое страницы программно через API нельзя.

### Вариант A — Переезд на Netlify (рекомендуется)
**Суть:** перенести хостинг статики с Tilda на Netlify, сохранив домен ghk-cr-track.ru.

**После переезда публикация = одна команда:**
```bat
netlify deploy --prod --dir=./7402
```
Или просьба к Claude Code: «опубликуй изменения».

**Что нужно сделать один раз:**
1. Зарегистрироваться на netlify.com (бесплатно)
2. Создать сайт, загрузить папку 7402/
3. Перевесить DNS: в панели регистратора сменить A-запись ghk-cr-track.ru → Netlify IP
4. `npm install -g netlify-cli` → `netlify login` → `netlify link`

**Плюсы:** бесплатно, 1 команда на все файлы, rollback одной кнопкой, не нужен браузер.
**Минусы:** один раз перевесить DNS (~15 мин).

---

### Вариант B — Playwright-автоматизация Tilda
**Суть:** скрипт логинится в tilda.cc, открывает редактор страницы, вставляет HTML, публикует.

**Плюсы:** не нужно менять хостинг.
**Минусы:** хрупко (ломается при изменении UI Tilda), нужно хранить пароль, ~30-60 сек на страницу, Tilda может заблокировать.

---

**Итог:** Вариант A значительно надёжнее и проще. Вариант B — временная мера если переезд невозможен.

## Приоритет реализации

1. **Сначала** — протестировать текущий лендинговый пайплайн на реальном эксперименте
2. **Потом** — добавить 5.1 (Jira) и 5.2 (База знаний) как отдельных агентов
3. **Затем** — 12 (Презентация) и 13 (Отчёт)
4. **В конце** — 7–8 (Чеклист после выката)
5. **Инфраструктура** — автоматизация публикации (Группа Г)
